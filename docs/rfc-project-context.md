## Summary

本 RFC 提出引入 `popsicle context scan` 命令，自动扫描项目文件生成技术画像，持久化为 `.popsicle/project-context.md`，并在 `popsicle prompt` 时作为背景上下文自动注入到所有 Skill 的 prompt 前部。这是对 domain priming 从 Skill 级别到项目级别的提升。

## Motivation

`popsicle prompt` 和 `popsicle context` 提供的是 **pipeline 运行时的文档上下文**——PRD、RFC、test-spec 等产出物。但 LLM 在执行 Skill 时还需要另一层信息：**项目本身的技术约束**——用什么语言、什么框架、什么编码规范、什么测试模式。

这类信息不属于任何 Skill 的产出物，却影响每个 Skill 的执行质量：

1. **架构决策偏离** — 一个 Rust 项目和一个 TypeScript 项目执行同一个 `rfc` Skill 时，架构决策空间完全不同。如果 LLM 不知道项目用 Rust，它可能建议一个依赖 Node.js 生态的方案。

2. **guide.md 的局限** — `guide.md` 提供通用的写作指导（"分析领域边界"），但缺少项目特定的技术约束。它是 Skill 级别的 domain priming，无法覆盖项目级别的差异。

3. **重复的手动补充** — 用户需要在每次会话中手动告诉 LLM "这是一个 Rust 项目，使用 Cargo workspace，CI 在 GitHub Actions 上"。这些信息是项目的 persistent ground truth，不应每次重复。

研究表明具体的领域约束（domain priming）比抽象的角色设定更能提升 LLM 性能（见设计哲学第 10 节）。Project context 把这个原则从 Skill 级别提升到项目级别。

## Proposal

### Detailed Design

#### 1. 项目扫描器（ProjectScanner）

新建 `popsicle-core/src/scanner.rs`，实现 `ProjectScanner` struct，通过纯文件系统检测（不执行构建命令）生成项目技术画像。

扫描器包含四个探测维度：

| 探测器 | 检测来源 | 提取信息 |
|--------|---------|---------|
| 语言探测 | `Cargo.toml`, `package.json`, `go.mod`, `pyproject.toml`, `pom.xml`, `build.gradle(.kts)` | 语言、版本、edition、关键框架 |
| 结构探测 | Cargo workspace members, npm workspaces, `pnpm-workspace.yaml`, 目录结构 | 项目类型（monorepo/单项目）、成员列表、notable 目录 |
| 工具链探测 | `.github/workflows/`, `.gitlab-ci.yml`, lint/fmt 配置文件, 测试框架配置 | CI、linting、formatting、testing 工具 |
| 依赖提取 | manifest 文件中的 dependencies | 关键依赖名称和版本 |

语言探测支持 5 种语言生态：

```rust
pub struct ProjectScanner { root: PathBuf }

impl ProjectScanner {
    pub fn new(project_root: &Path) -> Self;
    pub fn scan(&self) -> String;  // 返回完整 Markdown
}
```

每个 `detect_*` 方法返回结构化中间数据（`TechStack`、`ProjectStructure`、`DevPractices`、`Vec<Dependency>`），最终由 `render()` 统一渲染为 Markdown。

#### 2. 持久化存储

扫描结果写入 `.popsicle/project-context.md`，格式如下：

```markdown
# Project Context

> Auto-generated by `popsicle context scan`. You may edit this file freely.

## Tech Stack
- **Language**: Rust 1.75 (2024)
- **Frameworks**: tokio, serde
- **Build**: Cargo workspace

## Project Structure
- **Type**: Monorepo (Cargo workspace)
- **Members**: crates/popsicle-core, crates/popsicle-cli
- **Directories**: crates, docs, ui

## Development Practices
- **CI**: GitHub Actions
- **Linting**: clippy
- **Formatting**: rustfmt
- **Testing**: cargo test

## Key Dependencies
- clap 4.0
- serde 1
- rusqlite 0.32

## Notes
(Add project-specific context, conventions, or constraints here)
```

关键约束：
- **可编辑** — 自动生成后用户可手动补充和修改
- **不自动覆盖** — `scan` 命令检测文件已存在时拒绝覆盖，需 `--force` 显式重新生成
- **按需更新** — 不自动重新扫描，用户显式运行 `popsicle context scan` 时才更新

#### 3. Prompt 注入

在 `popsicle prompt` 的输出组装中，检查 `.popsicle/project-context.md` 是否存在，若存在则读取内容并放在最前面（low relevance 位置）：

```
## Project Context (background)
{project-context.md 内容}

---

## Input Context (from upstream skills)
{上游文档，按 relevance 排序}

---

{prompt 指令}
```

这与 Selective Context Injection（RFC-SCI）的 attention 优化策略一致：
- Project context 作为背景铺垫，放在 attention 最弱的前部
- 高 relevance 的上游文档和 prompt 指令放在末尾（attention 最强区域）
- Project context 不与上游文档竞争 attention，而是提供全局约束

#### 4. CLI 命令重构

将 `popsicle context` 从单命令重构为子命令组：

- `popsicle context show` — 原有的 pipeline context 输出（完全兼容）
- `popsicle context scan` — 新增，扫描项目并生成 `project-context.md`
  - `--force` 标志：覆盖已有文件

#### 5. Desktop UI 集成

在 Dashboard 页面新增可折叠的 Project Context 卡片：
- 默认折叠，标题行显示摘要（语言 · 构建系统）
- 展开后以 2 列网格展示各 section（Tech Stack / Project Structure / Development Practices / Key Dependencies / Notes）
- 每个 section 有对应图标
- 底部提示用户通过 CLI 更新

新增 Tauri command `get_project_context` 读取 `.popsicle/project-context.md` 并返回内容。UI 只读，不触发扫描。

### Interface Changes

#### CLI 命令变更

`popsicle context` 从扁平命令变为子命令组：

```bash
# 之前
popsicle context --run <id>

# 之后
popsicle context show --run <id>   # 原有功能
popsicle context scan              # 新增
popsicle context scan --force      # 覆盖已有
```

#### popsicle prompt JSON 输出

新增 `project_context` 字段：

```json
{
  "skill": "rfc",
  "state": "draft",
  "prompt": "...",
  "full_prompt": "...",
  "project_context": "# Project Context\n\n...",
  "input_context": "...",
  "context_parts": [...],
  "available_states": ["draft"]
}
```

#### ProjectLayout API

新增方法：

```rust
impl ProjectLayout {
    pub fn project_context_path(&self) -> PathBuf {
        self.root.join("project-context.md")
    }
}
```

#### Tauri API

新增 command：

```rust
#[tauri::command]
pub fn get_project_context(state: State<AppState>) -> Result<ProjectContextInfo, String>;
```

返回：

```typescript
interface ProjectContextInfo {
  available: boolean;
  content: string | null;
  path: string | null;
}
```

## Rationale and Alternatives

### Why This Approach

1. **最小侵入性** — 不引入新的配置模型或工作流概念。扫描器生成的是普通 Markdown 文件，prompt 注入逻辑仅在已有的 `build_full_prompt` 中增加一个条件分支。

2. **与 Popsicle 哲学一致** — 遵循 domain priming over persona（第 10 节）、Hybrid Storage（第 5 节，Markdown + 文件系统）、CLI Executes, UI Observes（第 2 节）的设计原则。

3. **可编辑性** — 自动扫描提供起点，用户可自由修改补充。比纯手动维护省力，比纯自动生成灵活。`## Notes` section 专为用户自定义预留。

4. **与 Selective Context Injection 协同** — Project context 作为 "low relevance background" 注入，与 RFC-SCI 的 attention 优化策略完全兼容，不需要额外的优先级机制。

### Alternative A: config.toml 声明式配置

在 `config.toml` 中新增 `[tech_stack]` section，由用户手动声明语言、框架等信息。

- 优点：不依赖文件系统探测，用户完全控制内容
- 缺点：需要用户手动填写和维护；TOML 格式不如 Markdown 对 LLM 友好；无法自动检测新增的工具或依赖

不选择此方案，因为自动扫描 + 可编辑 Markdown 提供了更好的开箱即用体验，同时保留了手动控制的灵活性。

### Alternative B: 每次 prompt 时实时扫描

不持久化，每次 `popsicle prompt` 时实时扫描项目文件生成技术画像。

- 优点：始终反映最新状态
- 缺点：每次调用增加 IO 开销；用户无法编辑或补充；扫描结果不可预测，难以调试

不选择此方案，因为项目技术画像变化频率极低（通常只在依赖升级或架构变更时变化），按需扫描 + 持久化 + 可编辑的策略更匹配实际使用模式。

### Alternative C: 嵌入 Skill guide.md

将项目技术信息嵌入每个 Skill 的 `guide.md` 中，通过模板变量注入。

- 优点：不需要新的注入机制
- 缺点：同一信息需要在所有 Skill 的 guide.md 中重复维护；guide.md 是 Skill 级别的通用指导，混入项目特定信息违反关注点分离

不选择此方案，因为项目级别的技术约束应有独立的存储和注入路径。

### Cost of Inaction

不实现此方案，LLM 在执行 Skill 时缺乏项目技术约束，可能建议与项目技术栈不匹配的方案。用户需要在每次会话中手动提供这些信息，这与 Popsicle "自动化工作流"的核心价值矛盾。

## Open Questions

- Questions to resolve during the RFC process:
  - 自动扫描应该提取哪些信息？当前实现覆盖 5 种语言生态和常见 CI/lint/test 工具。是否需要扩展到更多语言或工具？
  - 扫描器是否应递归扫描子 crate/package 的依赖？当前实现只读根目录的 manifest 文件。

- Questions to resolve during implementation:
  - Project context 是否应该参与 Guard 检查——如 RFC 提议的技术方案是否与 project context 声明的技术栈一致？
  - 是否需要支持用户在 `config.toml` 中声明技术栈，作为 scan 的补充或替代？
  - 200 行上限（与 Auto-Memory RFC 类似的约束）是否适用于 project context？
