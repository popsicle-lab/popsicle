# AI辅助编程对工程自动化的要求

# Abstract

LLM像一个有丰富技能对全能选手，但对环境（包括业务、工程）的感知以及结果把控性有限，我们想要掌握这一选手，使其供我们使用，必须提供丰富的环境能力，且约束其破坏行为。这与我们管理团队中新入职的疯狂技术天才一样，首先我们需要提供丰富的规格文档，让他知道我们项目的全貌，每一次行动前让他依据全貌主动检索足够的相关信息，其次我们需要将项目的工程自动化能力提高，让他可以快速感知到结果。

> 全文围绕这两条腿展开：**腿①「丰富的规格文档」让 AI 知道全貌**（对应「文档如何管理」「规格/意图驱动开发」），**腿②「工程自动化」让 AI 快速感知结果**（对应「自动化有哪些步骤」的质量、部署、可观测、验证验收）。两条腿缺一不可——只有规格没有自动化反馈，AI 是盲目射击；只有自动化没有规格，AI 不知道往哪打。

# 自动化有哪些步骤

## 代码管理

### 仓库管理

1. 尽管多仓库（Multirepo）有独立发布等优点，但 Monorepo 对 AI 和自动化最友好，原因如下：  
* 统一的上下文：AI 可以在一次会话中看到所有子产品代码，理解跨产品的依赖链，无需你手动解释“A 依赖 B 的哪个版本”。  
* 原子化变更：当需求涉及多个子产品时，一个提交就能包含所有改动，CI 可以自动验证整个依赖链的兼容性。  
* 一致的工具链：所有子产品共享构建、测试、语法检查配置，AI 学习的规则统一，不易出错。  
* 易于追溯与自动化：改动历史、问题追踪都在一个视图中，自动化的影响面分析更简单。  
2. 仅在以下情况考虑 Multirepo：  
* 子产品完全独立或领域划分清晰，有独立的安全边界和发布周期，且跨领域改动极罕见。  
* 你已有成熟的 API 版本管理和多仓库自动化编排系统（如基于 Git 子模块或包管理器），并对 AI 进行过专门的项目拆分训练。  
3. 那么单仓库的话如何组织  
* 清晰的目录划分，按产品边界组织  
* 边界明确：每个子产品有自己的 package.json/go.mod/Cargo.toml 等。  
* 共享库独立封装：库有独立的版本和测试，不直接在产品间复制代码。  
* 依赖声明与自动检测  
* 使用工作区工具声明依赖关系，如 npm workspaces、Yarn workspaces、Go workspace、Rust workspace、Gradle 多模块等。  
* 在 CI 中利用工具（如 Nx、Turborepo、Bazel、Lerna）进行依赖图分析，实现受影响模块的增量构建与测试。

### 分支管理

1. 为了让 AI 和自动化流水线以最高效的方式运作，放弃复杂的长期分支（如 Git Flow），使用干线开发。  
* 唯一长期分支：main (或 trunk)。  
* 极短命的特性分支：开发者在本地或远程创建一个与 main 同步的分支，存活时间通常不超过 1 天。  
* 所有代码必须通过自动化检查才能合并到 main。  
* 使用功能开关（Feature Flags） 处理未完成的半成品特性，而不是靠长期分支隔离。  
2. 为什么这对 AI 场景特别重要？  
* AI 生成代码时，上下文假设的是 main 分支的当前状态，分支越短，上下文越新，合并冲突越少。  
* Vibe coding 可能频繁“提 PR”，每次 PR 小而快，干线开发模式完美适配。  
* 功能开关允许 AI 大胆提交不完整代码，而不会阻塞发布，你可以在发布层面控制开关。  
3. 发布管理：  
* 不使用 release 分支，而是从 main 的特定提交打 tag（如 v1.2.3），自动化流水线按 tag 触发部署。  
* 如果需要长期维护多个版本（如 on-premise 部署），可创建 release-X.Y 分支，但只接受热修复，且修复必须先合入 main 再 cherry-pick 回去，保证主干始终是真理。

> 仓库管理（monorepo）与分支管理（干线）这两个选择，本质都在服务 AI 的同一个需求：**让它每次行动面对的现实都是「全部且最新」的**——monorepo 给「全部」，干线给「最新」。

## 质量管理

AI 生成代码的速度远超人工逐行 review 的速度，所以质量管理的核心命题变成：**如何用自动化手段，在不依赖人逐行读的前提下，判定一段 AI 代码是否可信。**

### 1. 测试门禁

* 覆盖率只能当「下限红线」，不能当「质量证明」。AI 完全有能力写出覆盖率 90% 但断言形同虚设（`assert True`、只调用不验证返回值）的测试。
* 测试的**可信度**靠三根支柱，而非覆盖率数字：
  * **变异测试（mutation testing）**：往源码里注入人为 bug，看测试是否会失败。测试抓不住注入的 bug = 测试无效。这是回答「AI 写的测试到底有没有用」的唯一客观手段。
  * **基于属性的测试（property-based testing）+ 不变量**：让框架自动生成大量输入，验证「无论输入如何，某不变量恒成立」。覆盖面远大于 AI 手挑的几个用例。
  * **关键路径强制人 review**：涉及钱、安全、不可逆操作的测试，必须人签字（见后文「关键位置要 review」）。
* 集成测试是否计入覆盖率：建议**分开统计**。单测覆盖率衡量「逻辑分支」，集成测试覆盖率衡量「跨模块路径」，混在一起会互相稀释、掩盖盲区。

### 2. 静态分析

* 作为**合并硬门禁**：lint、类型检查、安全扫描（SAST、依赖漏洞扫描）任一不过则 PR 阻塞。
* AI 代码尤其要开严格的静态规则——AI 倾向于「能跑就行」，静态分析是廉价且确定的第一道筛子。

### 3. 性能分析

* 关键位置生成 benchmark example，每次合并跑 CI 对比性能基线。
* 设回归阈值（如 P95 劣化 > 10% 即阻塞），避免 AI 在「让测试通过」的过程中悄悄引入性能退化。

## 自动化部署

* **干线打 tag 触发，零手工步骤**：接续分支管理——从 main 的特定提交打 tag，流水线按 tag 自动部署，不存在「某人手动点一下」的环节。
* **部署即代码（IaC）**：部署拓扑用声明式配置描述并纳入仓库，让 AI 能读懂「这个服务部署在哪、依赖什么」，而不是藏在某人脑子里或控制台点选里。
* **渐进发布是「让 AI 大胆提交」的安全网**：Feature Flag + 金丝雀/灰度发布，让不完整或高风险代码可以合并进 main 但不立即全量生效，把发布风险与代码合并解耦。
* **一键回滚是硬要求**：每次部署都必须可快速回退到上一个已知良好状态。AI 改动多、面广，回滚能力是兜底防线。

## 可观测性

可观测性是 Abstract 里「让 AI 快速感知结果」这条腿的**心脏**——没有它，AI 的每一次改动都是盲目射击。

* **不止三支柱，重点是「让 AI 能消费」**：logs / metrics / traces 必须结构化、可查询、可被 agent 在会话中直接拉取分析，而不只是给人看的仪表盘。
* **意图级遥测（intent-level telemetry）**：每一条 `acceptance.intent` 都应对应一个线上指标，持续监测「现实是否偏离意图」。这正是意图驱动开发里「意图验证与同步」在生产环境的落地形态。
* **反馈闭环**：把可观测数据回流给 AI 形成闭环——线上错误率、性能退化、用户行为，反向驱动文档 / 意图 / 代码的修订。文档读取热力图（哪些 spec 被 AI 高频检索、哪些从不命中）也是这个闭环的一部分。
* 具体采集与存储方案 `[TBD: 待技术栈确定]`。

## 验证验收

这一步可以说与质量管理有关联，可以放在质量管理里，但在 AI 编程领域，这一步是最后的防线，需要单独摘出。

它和质量管理的根本区别：质量管理回答「**做完了吗 / 做得规范吗**」，验证验收回答「**做对了吗**」——即产物是否真的满足最初的意图与契约。

* **形式化验证语言先生成，但暂不做验收门禁，慢慢观察效果**。早期让形式化规范（如 `.intent`）与 Z3 等求解器先跑起来、积累信号，但不阻塞合并，避免误报扼杀开发节奏。
* **从「观察」升级为「门禁」的退出判据**：当某个 product 的形式化检查连续 N 个迭代「判定结果 = 线上实际行为」零偏差后，再把它从观察模式切换为合并门禁。否则形式化验证会永远停在玩具阶段。
* **三道防线分层**：
  1. **意图一致性**（提交期）：`.intent` 之间是否自洽，用 SMT 求解器判定。
  2. **验收测试**（CI 期）：acceptance 条件是否被实现满足。
  3. **线上意图遥测**（运行期）：现实是否持续符合意图（见可观测性）。

## API协议版本管理

* **monorepo 内：契约即代码**。模块间接口用机器可读的契约（schema / `contracts.intent`）描述，消费方在**编译期 / CI 期**就能发现破坏，而不是等运行时炸。
* **跨边界（对外 / 跨仓库）：版本化 + 兼容性 CI**。破坏性变更必须显式升版本，并由 CI 自动检测「这次改动是否破坏了现有消费方」（接续仓库管理里的依赖图分析）。
* **AI 视角的硬要求**：API 契约必须机器可读，这样 AI 改动接口一端时，能自动知道波及哪些消费方、需要同步改哪里——这是 monorepo「原子化变更」能力的前提。

# 文档如何管理

## 产品文档与技术文档

文档按「视角」和「受众」分两类，措辞禁区互斥，但**都对 AI 开放**——区别在于它们回答不同的问题。

* **产品文档**（如 PRODUCT.md + 任务图）：用户视角与商业视角。Owner 是 PM；受众是 PM、销售、客户成功、AI Copilot。**禁止**写实现细节、技术选型理由。
* **技术文档**（如 ARCHITECTURE.md + contracts）：实现视角。Owner 是工程；受众是工程师、AI。**禁止**写商业策略、定价、客户分层。
* 两者都给 AI 检索，但服务不同问题：「用户能做什么」去问产品文档，「系统怎么实现」去问技术文档。互不污染，AI 才不会在回答产品问题时泄露实现细节、回答技术问题时被商业叙事干扰。

## 现状文档与演化文档

这是文档管理里最容易被忽视、却对 AI 检索质量影响最大的一条划分。

* **现状文档 = 活文档**：只描述「现在是什么样」。只有现在时，无版本号，**无历史叙事、无未来叙事**（不写「曾经用过 X」「未来计划 Y」）。过期内容就地修正。
* **演化文档 = 决策档案**（PDR / ADR）：只追加，永不修改，记录「为什么从 X 变成 Y」。
* **核心洞察**：把「历史」从活文档里彻底赶出去——历史是 `git log` + 决策档案的活。这样 AI 检索现状文档时，**不会被过期信息污染**，每一句话都是当前真理。
* **每次活文档编辑都要引用一个决策 ID**（错别字 / 链接 / 措辞修复除外）：保证活文档的每一处变化都能追溯到「为什么」，而那个「为什么」永久躺在演化文档里。

# AI编程的某些细节

## 关键位置要review

* **不是所有代码都 review**——否则 AI 提速的意义就没了。要做的是**定义「关键位置」清单**，只对清单内的改动强制人工签字。
* 关键位置通常包括：安全边界（认证 / 授权 / 加密）、涉及钱的逻辑（计费 / 结算）、不可逆操作（删除 / 数据迁移）、跨 product 契约、以及 invariants（不变量）的实现。
* 落地手段：用 CODEOWNERS + 路径/标签规则，让这些位置的改动**必须**指定人 review 才能合并；清单之外的改动靠自动化门禁（测试 + 静态分析 + 验收）兜底。

## 不要着急串联工作流，先测试好单个技能

* 把它写成硬纪律：**单个 skill 没有在 ≥ 1 个真实项目上达到质量基线之前，禁止把它接进 pipeline。**
* 每个 skill 应有自己的验收样本（输入 → 期望产出），单独测试通过再考虑编排。
* 串联的代价：一个 skill 的错误会跨 skill 放大，且在长链路里极难定位到底是哪一环出的问题。先保证每一环可靠，再谈自动化串联。

## 不同类型产品需要不同的技能和工作流

不要拿一套工作流套所有产品。至少按下面的维度分流：

* **业务 / CRUD 类**：IDD（意图驱动）足够——意图捕获、PRD 任务图、acceptance 验收这条链能覆盖正确性。
* **算法 / 仿真类**：IDD 用来管理**需求**的正确性，但实现的正确性需要**数学验证**。比如仿真需要 IDD，也需要**基于物理不变量的 metamorphic testing（变形测试）**：用 IDD 锁定「要算什么」，用 metamorphic testing 验证「算得对不对」——因为仿真往往没有可比对的「标准答案」，只能靠「输入变换后输出应满足某物理关系」来验证。
* **探索 / 不确定类**：需求本身还想不清楚时，先让 AI 实现一版、在实现基础上迭代修改（先实现后更改），而不是强行先写完整 spec。

# 规格驱动开发

规格驱动开发（SDD, Spec-Driven Development）：先把需求写成结构化规格（spec），再由 spec 驱动（人或 AI）生成代码与测试。

**SDD 与 IDD（意图驱动开发）的关键区别**必须划清，否则人和 AI 都会混淆：

* **SDD**：spec 是**中间产物**。目的是生成代码，代码生成完成后 spec 即可「躺平」，不再约束运行时。适合需求稳定、一次性交付的场景。
* **IDD**：意图是**长期资产**。代码上线后，意图继续约束运行时——通过意图遥测持续监测「现实是否偏离意图」（见「意图验证与同步」）。适合长生命周期、需要运行时持续保真的产品。

一句话：**SDD 的 spec 活到 codegen，IDD 的 intent 活到产品下线。**

本方法论整体采用的是 **IDD**：文档体系（活文档 + 决策档案 + 三层 intent）、任务图 PRD、acceptance / invariants / contracts 三类 intent，都是为了让意图成为贯穿「需求 → 实现 → 运行 → 演化」全生命周期的长期资产，而非用完即弃的脚手架。

# 意图驱动开发

## 意图驱动开发

1. 意图本体 (Intention Ontology)，AWISE 试图建立一个对人和机器都可读的意图分类体系，至少包含三层：  
* 用户意图 (User Intent)：例如“我想解析这堆 PDF 发票”。  
* 技术意图 (Technical Intent)：例如“需要 OCR 服务”、“提取6个字段”。  
* 过程意图 (Process Intent)：例如“先试用开源库，如果失败再尝试付费 API，整个过程要在5分钟内完成”。  
2. 意图驱动的代理工作流 (Agentic Workflow)：  
* 基于这个本体，他们设计了工作流模板，将意图分解为代理的任务图。每个任务节点都是一次“微观意图”，代理根据上下文自主规划具体行动，但边界由上层意图严格限定。  
3. 概念模型（5大核心活动）：  
* 意图捕获 (Capture)：从自然语言、用户操作、业务协议等提取意图。  
* 意图表达 (Expression)：将意图转化为结构化模型（如目标模型、模板、状态机、形式化规范）。  
* 意图精炼 (Refinement)：将高层意图分解为子意图或可执行策略，处理冲突。  
* 意图执行 (Execution)：系统根据当前状态自动规划、调度动作以达到意图。  
* 意图验证与同步 (Verification & Synchronization)：运行时持续监测现实是否与意图偏离，并自我调整。  
4. 意图重建与代码生成  
* 意图模型构建：  
* 用基于规则和 WordNet 的 NLP 流水线，从自然语言需求中抽取 意图元素（Intent Elements）：  
- 意图对象 (Object)：被操作的概念实体。例如“账户余额”。  
- 意图操作 (Operation)：对对象的动作。例如“更新、计算、验证”。  
* 同时，用程序静态分析从代码中提取对应的元素（如方法调用、字段访问）。  
    
* 意图相似度计算  
* 将需求和代码的意图元素映射到一个词汇-语义空间，计算它们之间的意图相似度分数。这里结合了信息检索（IR）和语义知识。  
    
* 可追溯性链接建立  
* 通过意图相似度矩阵，自动生成从需求到代码片段的映射链接，并给出置信度。

## 意图驱动软件

1. 核心洞见：功能导向 vs. 意图导向

传统软件是“功能导向”的：开发者预制所有功能，用户通过菜单、按钮触发。意图导向软件则设想：

* 用户只表达意图（“帮我规划一个为期三天的巴黎艺术之旅”）。  
* 软件没有固定形态，而是根据意图动态组装一组临时渲染的UI组件、服务调用和代理协作流。  
2. 提出的两种互补架构  
* 架构A：极简即时应用 (Minimalist Instant Apps)  
  在大模型基础上，每个意图都能生成一个一次性、专门化的微型应用界面。这种应用包含：意图解析器 → 任务规划器 → 动态UI渲染器 → 结果汇总器。用完即焚，不留固化的功能代码。  
* 架构B：代理社会 (Society of Agents)  
  多个异构 AI 代理（旅行代理、日历代理、预算代理）构成一个社会，通过“意图广播”和“意图拍卖”机制，自主协商、分工完成一个复杂意图。人更像一个“市长”而非“操作员”。

# 已有项目分析

已有项目迁移方案：

1. 使用"绞杀者"模式（Strangler Fig）在原有项目上原地迁移  
2. 使用"绞杀者"模式（Strangler Fig）向新项目逐步迁移  

**两种模式的选择判据：**

* 选**原地迁移**：代码库整体健康、构建 / 测试基础设施可用、团队对历史有把握、不希望中断现有发布节奏。风险低、改动可控，但受限于旧结构。
* 选**迁到新仓库**：旧项目技术栈 / 目录结构已严重阻碍 AI 与自动化、构建链难以挽救、或想借机重铺 monorepo + 干线 + IDD 文档体系。回报高，但迁移期需同时维护新旧两套，成本与风险更大。
* 无论哪种，绞杀者模式的核心都是：新老并存、按切片逐步替换、每替换一块就用自动化验证等价性，直到旧系统被完全「绞杀」。迁移工作流的选择应作为用户偏好固化（见 Popsicle 偏好设置）。

# 技能

> 每个技能都应标注成熟度状态（实验中 / 单测通过 / 真实项目验证过）。未达「真实项目验证过」的技能不得接入自动化 pipeline（见「不要着急串联工作流」）。

产品PRD工作流

* 产品文档Spec：  想法 → PRFC → 评审 → PDR (Accepted) → PRODUCT.md 更新  
* 技术文档Spec：  想法 → RFC → 评审 → ADR (Accepted) → ARCHITECTURE.md 更新  
* 先实现后更改：有时候我们不知道应该把一个产品做成什么样，想要让AI先实现，在实现基础上更改  
* 开发合并：每一次改动都要新建分支，完成后将主分支和到开发分支，并通过所有CI check，才可以合并到主分支  
* 提供两种迁移工作流，一种是将老项目迁移到新仓库，另一种是在原仓库原地更新  
* PRD应该是以任务和意图为中心，告别功能罗列

## Popsicle

* 提供用户偏好设置，偏好设置更新到[Cluade.md](http://Cluade.md)或Agent.md，偏好包括用户输入输出语言、老项目迁移是新仓库还是原地等  
* Popsicle默认使用能听懂人话的LLM，评分中等及以上，并且只测试相关模型，不会浪费时间去适配低端模型  
* 提供trace，比如文档读取热力图
