# intent-lang 定位声明（POSITIONING）

> 本文档明确 intent-lang 在软件工程体系中的定位边界。
> 是 intent-lang 所有设计决策的最高准绳——任何语言、工具、文档变更必须与本文档一致。

---

## 一句话定位

**intent-lang 是一种需求建模 DSL（Requirements Modeling DSL）：用来把自然语言需求和领域不变量翻译成机器可读、可形式化验证的"意图"，并通过 Z3 验证多份意图之间无逻辑矛盾。**

---

## 它**是**什么

| 维度 | 定位 |
|---|---|
| **语言类别** | 需求建模 DSL（Requirements Modeling DSL） |
| **核心用途** | 把"业务/领域意图"翻译成"机器可读的逻辑断言" |
| **形式化层级** | L2（SMT 求解），不是 L4（定理证明） |
| **验证目标** | 意图集合的**一致性**（无矛盾）+ **完备性**（无遗漏） |
| **下游消费者** | LLM、popsicle Skill、人工评审、测试规格生成器、API 契约校验器 |

intent-lang 的产物（`.intent` 文件）是**所有下游行为的可信锚点**：
- LLM 基于 intent 生成实现代码 → 实现是否符合 intent，由 intent 反向校验
- LLM 基于 intent 生成测试规格 → 测试覆盖率以 intent 为基准
- 人写 intent → 用 Z3 验证不和已有 intent 矛盾
- 决策记录（PDR/ADR）变更 → 触发 intent diff 与影响分析

---

## 它**不是**什么

为避免误用与定位漂移，明确以下边界：

### ❌ 不是实现语言
intent-lang **不生成可执行代码**。
intent 不描述"怎么做"，只描述"做到什么程度算对"。
代码生成是 popsicle Skill / LLM Agent 的职责，不是 intent-lang 的职责。

### ❌ 不是程序规格语言（不要用它替代 Dafny / Why3）
程序规格语言关注的是 **"实现是否满足规格"**——例如 Dafny 给一个排序函数加 `requires`/`ensures` 然后证明实现正确。
intent-lang 关注的是 **"规格自身是否自洽、是否完整、是否反映业务意图"**——更靠近需求与决策侧。

| 问题 | 谁回答 |
|---|---|
| "这个排序算法实现得对吗？" | Dafny / Why3 |
| "我们到底想要什么样的排序？稳定？原地？时间复杂度上限？这些约束之间有矛盾吗？" | intent-lang |

### ❌ 不是通用契约语言（不直接约束 API 调用）
意图是**业务规则的形式化**，不是**单次 API 调用的前后断言**。
API 契约可以从 intent 派生（由 popsicle Skill 完成），但 intent 文件本身不应成为 API schema 的代名词。

### ❌ 不是建模语言 / 程序设计语言（不替代 TLA+ / Alloy 设计算法）
TLA+ / Alloy 用于**设计并验证算法/协议的正确性**。
intent-lang 不关心算法层面的死锁、活锁、分布式状态机。
如果你的需求是"设计一个 Paxos 变种"，请用 TLA+；intent-lang 的层次更高，描述的是"我们对一致性的业务承诺"。

### ❌ 不是 AI 优先语言
intent-lang 的语法**先服务于人类可读性**，AI 友好是副产品。
人写出来的 intent 必须是同事一眼能看懂的；LLM 写的 intent 必须能被人审查、评审、修正。
任何"为了 LLM 方便而牺牲人类可读性"的语言决策都是错的。

## 在 IDD 体系中的位置

```
┌─────────────────────────────────────────────────────────┐
│  人类需求（自然语言 / PRD / RFC / ADR / 用户访谈）       │
└────────────────────────┬────────────────────────────────┘
                         ↓ 翻译（人 + LLM 协作）
┌─────────────────────────────────────────────────────────┐
│  intent-lang  ←  这一层是 intent-lang 的核心战场         │
│  - 形式化意图（goal / invariant / theorem / safety）    │
│  - Z3 验证一致性（无矛盾）                              │
│  - 工具检测完备性（无遗漏）                             │
└────────┬────────────────────────────────────┬───────────┘
         ↓ 派生（popsicle Skill）            ↓ 反向追溯
┌────────────────────┐  ┌────────────────────────────────┐
│ 测试规格 / 测试代码  │  │ API 契约 / 实现代码             │
└────────────────────┘  └────────────────────────────────┘
```

intent-lang 在体系中扮演 **"可信锚点"** 的角色：上游对接需求与决策，下游对接代码与测试，向上向下都用 SMT 验证保证一致。

---

## 三大核心能力

### 1. 一致性验证（Consistency）—— 当前已有
> 多份意图之间不能存在逻辑矛盾。

例：
- intent A 要求 `account.balance >= 0`
- intent B 在某条路径下导致 `account.balance == -10`
- → Z3 找到反例，验证失败

### 2. 完备性检测（Completeness）—— 规划中
> 业务声称要覆盖的场景维度，是否都有对应的 intent。

例：业务说要支持 `[CNY, USD, EUR] × [转入, 转出, 退款] × [普通账户, 冻结账户]`，工具自动检测哪些组合**没有任何 invariant 覆盖**。

注意：完备性永远不可能 100%，工具的目标是给"已知未覆盖清单"。

### 3. 影响分析（Impact Analysis）—— 规划中
> 一次 intent 修改的影响范围。

例：放宽 `balance >= 0` 为 `balance >= -overdraft_limit`，工具自动列出：
- 受影响的下游 intent
- 受影响的测试规格
- 已标注追溯链的代码位置

## 设计纪律

任何 intent-lang 的语言/工具/文档变更，必须先回答：

1. **它服务于"需求建模"还是"实现/程序规格/建模设计"？** 后者一律拒绝。
2. **它是先解决人类可读性还是 LLM 可读性？** 先解决人类可读性。
3. **它是否经过真实场景 dogfood 验证？** 没有就先放进规划，不进语言层。
4. **它是否与 popsicle / Z3 的能力边界清晰？** 不抢它们的活。

## 与其他形式化工具的差异化矩阵

| 工具 | 关注层 | 用户写什么 | 验证什么 |
|---|---|---|---|
| **intent-lang** | 需求/意图层 | 业务规则、领域不变量 | 意图之间无矛盾、覆盖完备 |
| Dafny / Why3 | 程序规格层 | 函数前后断言 | 实现满足规格 |
| TLA+ / Alloy | 系统设计层 | 状态机、协议 | 算法/协议无活锁/死锁 |
| Lean / Coq | 数学/定理层 | 定理与证明 | 数学命题成立 |
| TypeScript | 类型层 | 类型签名 | 类型一致 |

**关键差异**：intent-lang 比这些都更靠"上游"——它关心的是"我们到底要做什么、要什么、不要什么"，而不是"实现/算法/类型对不对"。

## 与 popsicle 的协作分工

| 职责 | 归属 |
|---|---|
| 写 intent | 人 + LLM（在 popsicle Skill 引导下） |
| 验证 intent 一致性 | intent-lang（Z3） |
| 验证 intent 完备性 | intent-lang（工具） |
| 把 intent 翻译成测试规格 | popsicle Skill（intent-to-testspec） |
| 把测试规格翻译成红测试代码 | popsicle Skill（testspec-to-redtest） |
| 让红测试变绿（写实现） | popsicle Skill + LLM（red-to-green） |
| 突变测试验证测试有效性 | popsicle Skill（mutation-verifier） |
| 维护 intent ↔ 代码追溯链 | popsicle Skill（intent-trace-annotator） |
| 决策档案 / 活文档 | popsicle Skill |

**intent-lang 只做最核心的事：把意图变成可验证的逻辑断言，验证它们之间不矛盾。** 其他事不做。
