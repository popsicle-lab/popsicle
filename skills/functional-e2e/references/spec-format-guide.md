# Spec 格式规范 — 自然语言功能测试 Spec

## 设计原则

1. **人能直接读懂** — 非技术人员也能理解测试意图，无需对照代码
2. **精确到可执行** — 验证要点写到可以直接翻译为 assert 语句
3. **追溯来源** — 每个场景标注来自哪个需求或架构决策
4. **自包含** — 不需要对照其他文档就能理解场景完整语义

## 文件结构

每个 Spec 文件对应一个业务领域（与 `domain-map.md` 中的领域对应）。

```yaml
领域: [领域名称]
概述: [一句话描述本 spec 覆盖的测试范围]
适用部署: 全部 | SaaS | 独立部署

场景:
  - 名称: ...
    视角: POS | BND | CHN | INV | LFC | DPL
    优先级: P0 | P1 | P2
    背景: ...
    前置条件: [...]
    操作步骤: [...]
    验证要点: [...]
```

## 字段说明

### 领域 / 概述 / 适用部署

文件级别的元信息。`适用部署` 表示本 Spec 整体适用于哪些部署形态，场景级别可以单独覆盖。

### 场景

一个 Spec 文件包含多个场景，每个场景是一个独立的测试用例或测试组。

---

### 名称

用自然语言描述测试目的，不用代号。

- 好: "新租户注册后自动获得默认配额"
- 差: "S1-AC4-quota-init"

### 视角

六选一（POS / BND / CHN / INV / LFC / DPL），标明本场景的测试聚焦点。

### 优先级

P0 / P1 / P2，含义见 SKILL.md 的 Priority Strategy 段。

### 背景

2-3 句话说明：
- 为什么要测这个？
- 来自哪个需求或架构决策？
- 这个场景的风险是什么？

### 前置条件

测试开始前环境必须满足的条件。用自然语言描述，不写代码。

好的写法:
```yaml
前置条件:
  - 已创建两个独立租户，各自有管理员账号和已注册设备
  - 平台认证服务和数据服务已启动
```

差的写法:
```yaml
前置条件:
  - tenant_factory.create('a')
  - grpc.channel_ready_future(channel).result(timeout=5)
```

### 操作步骤

按顺序描述每一步的操作和预期结果：

```yaml
操作步骤:
  - 步骤: [做什么——精确到调用什么接口或执行什么操作]
    预期: [期望发生什么——精确到返回值、状态变化]
    捕获: [如果后续步骤需要用到本步骤的输出，说明保留什么]

  - 步骤: [下一步]
    预期: [预期结果]
```

**步骤**的精确度要求：读者看完就知道调用哪个接口、传什么参数。
**预期**的精确度要求：写到可以翻译为 assert 的程度。
**捕获**是可选字段：只在后续步骤需要引用本步骤输出时才写。

### 验证要点

场景结束后必须通过的关键断言。这是整个场景的核心交付物。

好的写法:
```yaml
验证要点:
  - 返回的组织ID为正整数，项目ID为正整数
  - 访问令牌是非空字符串，可以被解析为合法 JWT
  - 数据在认证服务和数据服务之间传递过程中保持完整性
```

差的写法:
```yaml
验证要点:
  - 注册成功
  - 能正常使用
```

### 关联需求（可选）

来自 PRD / ADR 的原始需求引用，方便追溯。

```yaml
关联需求:
  - "SaaS 新租户 onboarding 时间 < 5 分钟"
  - "ADR-0010: JWT 优先于 Header"
```

## 各视角的特殊字段

### 边界条件（BND）— 测试矩阵

当场景涉及多个变化维度的组合测试时，使用"测试矩阵"段。

```yaml
测试矩阵:
  变化维度:
    - 调用者身份:
        - 本租户合法令牌 — 正常的认证凭证，组织归属正确
        - 其他租户令牌 — 认证凭证有效但属于另一个组织
        - 伪造请求头 — 合法令牌但手动篡改组织ID请求头
        - 无认证 — 不携带任何凭证

    - 目标资源归属:
        - 全部自有 — 操作目标都属于调用方
        - 全部他方 — 操作目标全属于其他租户
        - 混合归属 — 部分自有部分他方（验证原子性）

  排除组合:
    - 无认证时角色区分无意义，只需验证一种"无认证"情况
    - 只读角色被权限拒绝即可，不需穷举目标归属的每种情况

  预估测试点: 约 25 个
```

关键要求：
- 每个维度取值后面用 " — " 跟上自然语言解释
- 排除组合用自然语言说明为什么排除
- 预估测试点 = 笛卡尔积 - 排除项

### 安全不变量（INV）— 护城河断言

```yaml
关联不变量: INV-04
护城河断言: >
  批量写入 N 个设备时，若任一设备不属于当前租户，
  整个请求被拒绝，已校验通过的设备也不写入，写入计数为零
```

护城河断言的精确度：必须能直接翻译为单个 assert 语句或一组强耦合的 assert。

### 状态生命周期（LFC）— 状态转换链

```yaml
状态转换:
  - 从: 未注册
    触发: 调用设备注册接口
    到: 已注册
    断言: 设备出现在设备列表中，状态为已注册

  - 从: 已注册
    触发: 首次上报属性数据
    到: 活跃上报中
    断言: 查询设备状态返回上报的属性值

  - 从: 活跃上报中
    触发: 再次上报属性数据
    到: 活跃上报中
    断言: 属性值更新为最新上报值
```

### 部署形态（DPL）— 对偶矩阵

```yaml
对偶矩阵:
  - 功能: 自助注册
    SaaS: 返回成功，自动创建组织和项目
    独立部署: 返回 UNIMPLEMENTED

  - 功能: 组织管理
    SaaS: 支持创建和查询多个组织
    独立部署: 返回 UNIMPLEMENTED，使用默认组织
```

## 完整示例

```yaml
领域: 租户数据隔离
概述: 验证不同租户之间的数据完全隔离，跨租户操作不泄漏任何信息
适用部署: SaaS

场景:
  - 名称: 租户A的数据对租户B完全不可见
    视角: CHN
    优先级: P0

    背景: >
      多租户 SaaS 平台的核心安全要求是租户间数据完全隔离。
      验证一个租户创建的物模型和设备数据，在另一个租户的视角下完全不存在。
      来自 PRD 成功指标"租户数据泄漏事件=零"。

    前置条件:
      - 创建两个独立的租户A和租户B，各自有管理员账号
      - 平台认证服务和数据服务已启动

    操作步骤:
      - 步骤: 租户A的管理员登录并创建一个物模型，定义温度和湿度两个属性
        预期: 物模型创建成功
        捕获: 物模型ID

      - 步骤: 租户A在该物模型下注册一台设备并上报一组温度数据
        预期: 设备注册成功，数据上报成功

      - 步骤: 切换到租户B的管理员身份，尝试查询租户A创建的物模型
        预期: 返回"未找到"或空结果，不返回权限错误

      - 步骤: 租户B尝试列出所有设备
        预期: 返回的设备列表中不包含租户A的设备

      - 步骤: 租户B尝试直接用租户A的设备ID查询设备状态
        预期: 返回"未找到"，不暴露该设备存在于其他租户

    验证要点:
      - 跨租户查询返回统一的"未找到"而非"权限不足"（防止存在性推测）
      - 不返回服务器内部错误（500）
      - 租户B的操作不影响租户A的数据完整性

    关联需求:
      - "PRD: 租户数据泄漏事件=零"
      - "ADR-0010: 跨租户零信息泄漏"

  - 名称: 跨租户请求伪造防御
    视角: BND
    优先级: P0

    背景: >
      攻击者可能使用合法令牌但篡改请求头中的组织ID来尝试访问其他租户数据。
      根据 ADR-0010 的架构决策，系统必须以 JWT 中的信息为准，忽略请求头。

    前置条件:
      - 创建两个独立租户，各自有管理员账号和已注册设备

    操作步骤:
      - 步骤: 租户A登录获取合法令牌（JWT 中 org_id=A）
        预期: 登录成功
        捕获: 租户A的访问令牌

      - 步骤: 使用租户A的令牌，但在请求头中将组织ID改为租户B的值，发起数据操作
        预期: 系统忽略篡改的请求头，按JWT中的组织ID处理

    验证要点:
      - 系统使用 JWT 中的组织ID，忽略请求头中的值
      - 操作结果与不篡改请求头时完全一致
      - 不会因为请求头不一致而返回内部错误

    测试矩阵:
      变化维度:
        - 调用者身份:
            - 本租户合法令牌 — JWT 中的组织与实际操作目标一致
            - 其他租户令牌 — JWT 属于另一个组织
            - 伪造请求头 — JWT 合法但请求头中的组织ID被篡改为其他值
            - 无认证 — 不携带任何凭证

        - 操作类型:
            - 读取操作 — 查询设备状态、列出设备
            - 写入操作 — 上报属性、创建物模型
            - 管理操作 — 注册设备、删除物模型

      排除组合:
        - 无认证时操作类型区分无意义，只需验证被拒绝
        - 本租户合法令牌属于正向功能验证（POS 视角），此处只关注攻击路径

      预估测试点: 约 8 个

    关联不变量: INV-01
    护城河断言: >
      当请求头中的组织ID与JWT中的组织ID不一致时，
      系统必须以JWT为准，忽略请求头值，并记录告警日志
```
