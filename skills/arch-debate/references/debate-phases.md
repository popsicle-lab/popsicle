# 辩论阶段详细说明

辩论共分 4 个阶段，每阶段有明确的参与角色、活动内容、暂停点和产出。

---

## Phase 1: 问题定义与约束澄清

**目标**: 确保所有参与者对问题空间有共同理解

**主要角色**: ARCH（主持） + 领域专家 + 约束专家各 1 位

**活动内容**:

1. ARCH 陈述对问题的理解，列出：
   - 需要决策的核心问题
   - 已知的约束条件（技术、业务、资源）
   - 决策的影响范围

2. 领域专家补充：
   - 领域特有的约束和惯例
   - 类似系统的常见做法
   - 关键的技术限制

3. 约束专家补充：
   - 非功能性要求（安全、性能、可用性）
   - 运维和部署约束
   - 合规和数据治理要求

**暂停点**（1 次）:
```
🎤 Phase 1 暂停
📋 问题定义如上。以下是已识别的关键约束：
   1. [约束 1]
   2. [约束 2]
   3. [约束 3]
❓ [根据置信度调整的提问]
```

**阶段产出**:
- 问题陈述（1-2 句话）
- 约束清单（必须满足 / 最好满足 / 可以取舍）
- 成功标准（如何判断方案是好的）

---

## Phase 2: 方案发散与初评

**目标**: 产出 2-3 个差异化的候选方案

**主要角色**: 2-3 位角色分别提出不同方案

**活动内容**:

1. 角色分配：
   - 选择 2-3 位风格偏好不同的角色，每人独立提出一个方案
   - 方案之间应有实质性差异（不同架构风格、不同技术选型、不同权衡取向）

2. 每个方案包含：
   - 方案名称和一句话概述
   - 核心架构图/伪代码（文本描述）
   - 主要技术选型及理由
   - 自评优势和劣势

3. 初步交叉评论：
   - 每位提案者简短评论其他方案的最大优势和最大风险

**方案差异化指导**:

确保方案之间存在真正的架构差异，而非细节差异。典型的差异维度：

| 维度 | 方案 A 方向 | 方案 B 方向 |
|------|-----------|-----------|
| 一致性 vs. 可用性 | 强一致（CP） | 最终一致（AP） |
| 简单 vs. 灵活 | 单体/简单实现 | 微服务/可扩展 |
| 自建 vs. 依赖 | 自研核心组件 | 引入成熟第三方 |
| 推 vs. 拉 | 推送模型 | 拉取/轮询模型 |
| 预计算 vs. 即时 | 物化视图 | 实时查询 |

**暂停点**（1 次）:
```
🎤 Phase 2 暂停
📋 当前有 [N] 个候选方案：
   方案 A: [一句话概述]
   方案 B: [一句话概述]
   方案 C: [一句话概述]（如有）
❓ [根据置信度调整的提问]
```

**阶段产出**:
- 2-3 个结构化的候选方案
- 各方案的自评优劣势
- 初步的交叉评论

---

## Phase 3: 多角色辩论

**目标**: 从多维度深度评审每个候选方案，暴露风险和盲点

**主要角色**: 全部参与角色（除已提方案的角色外的评审角色）

**活动内容**:

1. 逐角色评审：
   - 每位评审角色从自己的视角评估所有候选方案
   - 指出各方案在自己关注领域的优劣
   - 给出明确的偏好排序及理由

2. 评审格式：
   ```
   **[角色ID - 角色名称]** 评审:

   方案 A:
   - ✅ [优势点]
   - ⚠️ [风险/不足]

   方案 B:
   - ✅ [优势点]
   - ⚠️ [风险/不足]

   **偏好**: 方案 [X]，因为 [理由]
   ```

3. 辩论环节：
   - 观点冲突时，涉及的角色互相回应
   - 利用角色间的内在张力推动深入讨论
   - 不回避分歧，而是明确记录分歧点及各方理由

**暂停点**（1-2 次）:

每 1-2 个角色评审后暂停一次：
```
🎤 Phase 3 暂停
📋 [角色 X] 评审完毕。主要观点：[要点]
❓ [根据置信度调整的提问]
```

**阶段产出**:
- 每位角色的评审意见
- 识别出的关键分歧点
- 各方案的综合优劣势对照

---

## Phase 4: 收敛与决策

**目标**: 形成最终推荐方案，产出可行动的架构文档

**主要角色**: ARCH（主持收敛）+ 全部角色投票

**活动内容**:

1. ARCH 综合所有评审意见：
   - 梳理各角色的偏好和理由
   - 识别共识点和分歧点
   - 提出综合推荐（可以是某方案 + 其他方案的局部优点）

2. 角色投票：
   - 每位角色投票：支持 / 有保留地支持 / 反对
   - 反对者必须说明具体顾虑和接受条件

3. 用户最终决策：
   - 向用户展示投票结果和分歧
   - 请用户做最终决定
   - 对于用户推翻多数角色意见的情况，标注为"用户决策覆盖"

4. 产出生成：
   - 基于最终决策生成 ADR 草稿、会议纪要和决策矩阵
   - 模板参见 `output-templates.md`

**暂停点**（1 次）:
```
🎤 Phase 4 最终确认
📋 投票结果：
   方案 A: [N] 票支持 — [角色列表]
   方案 B: [N] 票支持 — [角色列表]
   分歧: [角色 X] 强烈反对方案 A，因为 [原因]
🏆 ARCH 推荐: 方案 [X]，并采纳方案 [Y] 的 [具体部分]
❓ 你同意这个结论吗？需要调整吗？
```

**阶段产出**:
- 最终推荐方案（含修正和折中）
- ADR 草稿
- 会议纪要
- 决策矩阵

---

## 阶段小结格式

每个 Phase 结束时生成阶段小结：

```markdown
---
### Phase [N] 小结

**共识**:
- [共识点 1]
- [共识点 2]

**用户决定**:
- [用户做出的关键决定]

**悬而未决**:
- [待下阶段讨论的问题]

---
```

## 异常处理

**讨论陷入僵局时**:
- ARCH 角色介入，将分歧拆分为更小的子问题
- 对子问题逐一达成共识
- 如仍无法收敛，标记为"需要更多信息"并列出所需信息

**用户长时间不回应暂停点时**:
- 假设用户选择"继续"，进入下一步
- 在阶段小结中标注"此暂停点用户未参与"

**讨论偏离主题时**:
- ARCH 角色提醒并拉回主线
- 将偏离话题记录为"后续讨论项"
