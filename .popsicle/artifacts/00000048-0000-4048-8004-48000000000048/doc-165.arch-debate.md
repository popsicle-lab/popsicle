# Phase D report 架构辩论

> topic: WAL 只读聚合 vs OTLP 查询；report 模块归属

## Topic

Phase D：`action=report` 从本地 WAL 聚合 run 指标，是否扩展 `crates/telemetry` 还是新 crate？

## Participants

| 角色 | 立场 |
|---|---|
| ARCH | 扩展 telemetry crate，report.rs 只读 WAL |
| SEC | 报告不含 token 原文；沿用 redaction |
| PERF | 大 WAL 线性扫描可接受 MVP；P1 分页 |
| OPS | 本地 first；OTLP 仪表盘互补 |
| DEV | cli-ux 仅转发 tool run |

用户置信度：4/5

## Phase 1 — 技术问题 + NFR

- 问题：把 span 变成维护者可读的 RunReport JSON
- 硬约束：fail-open、不进 state.db、ADR-001 sidecar
- NFR 优先级：可用性 > 性能 > 可演进性

## Phase 2 — 方案发散

- **A** telemetry/report.rs 读 WAL — 推荐
- **B** 新 crate telemetry-report — 过度拆分
- **C** 仅 OTLP 查询 — 无 endpoint 不可用

## Phase 3 — 多角色评审

| 方案 | ARCH | SEC | 结论 |
|---|---|---|---|
| A | ✅ | ✅ | **选用** |
| B | ⚠️ | ✅ | 否决 — crate 膨胀无收益 |
| C | ⚠️ | ✅ | 否决 — 离线不可用 |

## Phase 4 — 决议

1. `crates/telemetry/src/report.rs` 只读解析 WAL
2. `action=report` 与 record/flush/status 同入口
3. weekly 摘要由 living-doc-author 写 PROJECT_CONTEXT，不在 report 内写文件
4. → **ADR-002** 候选
