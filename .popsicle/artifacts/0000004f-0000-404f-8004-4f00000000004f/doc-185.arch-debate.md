# 架构辩论纪要 — agent-runtime 执行面分层

## Topic

agent-runtime 的 **Server / Daemon / popsicle CLI / Agent CLI** 如何切 crate 与状态真相源？来源：PDR-001 § Intent Mapping `contracts` 行与 doc-184 §6 行 6。

## Participants

| 角色 | 立场速写 |
|---|---|
| ARCH | 新建 agent-daemon + server crate，cli-ux 只暴露 daemon 子命令 |
| SEC | 密钥与 git 写操作仅 Daemon 进程；Server 存 task 元数据无 prompt 全文 |
| PERF | Daemon 并发上限对齐 Multica 类比；state 以本地 SQLite 为准 |
| OPS | Docker compose 自托管；telemetry 沿用 fail-open |
| DEV | subprocess 调现有 popsicle binary，不重写 pipeline 状态机 |

用户置信度：4/5

## Phase 1 — 技术问题 + 质量属性

- **要解决的问题**：远程派活如何反转调用链且不重写 skill-runtime 状态机。
- **硬约束**：ADR-007 cli-ux 为 IO shell；skill-runtime 拥有 pipeline session；charter Layer Map。
- **NFR 优先级**：安全 > 可演进性 > 可用性 > 性能（P0）。
- **事实基**：F-1 products/cli-ux/ARCHITECTURE.md — cli-ux 不拥有状态机；F-2 PDR-001 contracts 待 ADR。

## Phase 2 — 方案发散

- **方案 A（ARCH）**：`crates/agent-daemon` + `crates/agent-server`（或 monorepo 名 `remote-api`）；cli-ux 增 `daemon` 子命令；Mobile 调 Server REST+WS。
- **方案 B（DEV）**：全部塞进 `cli-ux` 一个 crate，无独立 server。
- **方案 C（OPS）**：Server 复用 legacy popsicle-sync WebSocket（已砍）+ 外挂 Multica。

## Phase 3 — 多角色评审

| 方案 | SEC | PERF | OPS | DEV |
|---|---|---|---|---|
| A | Server 无密钥 ✅ | 本地 SQLite 主路径 ✅ | 新 compose 可控 ✅ | subprocess 复用 ✅ |
| B | 壳臃肿 ❌ | 难测 ❌ | 单二进制耦合 ❌ | 违反 ADR-007 精神 ❌ |
| C | 双栈 ❌ | 未知 ❌ | 依赖外部 ❌ | IDD 不对齐 ❌ |

## Phase 4 — 收敛与决策

- **ARCH 综合**：方案 A；state 真相在 `.popsicle/state.db`，Server 持 task/run **镜像** + 事件流。
- **投票**：ARCH/SEC/PERF/OPS/DEV 支持 A；否决 B/C。
- **用户决策**：接受 A；CADR-001 限定 IDD-only 派活。

## Decision

采用 **三层执行架构**：agent-server（协调）+ agent-daemon（本机执行）+ subprocess `popsicle`/`cursor-agent`；cli-ux 仅增加 `daemon` 命令族转发至 agent-daemon 库。

## 关键分歧

- **状态镜像 vs 单一真相**：DATA 主张 Server 可写 state — 否决；仅 Daemon 写本地 state，Server 异步收事件 `[待验证: P0 单向同步]`。
- **charter**：CADR-001 将 agent-runtime 标为 IDD 派活扩展，非通用 Jira。
