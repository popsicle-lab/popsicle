# 产品辩论纪要 — Agent Runtime 手机远程派活

> **Status**: Draft (from product-debate skill)
> **Date**: 2026-07-06
> **Target Product**: `agent-runtime`
> **Input Mode**: `greenfield-product-brief`
> **User Confidence**: 4/5
> **Participants**: PM, UXR, GROWTH, ENGLD, BIZ

---

## Topic

是否为 Popsicle 新增 **agent-runtime** product，使开发者能通过手机远程派活，由本机 Daemon 自动执行 `popsicle` pipeline 并调起 Agent CLI 完成 IDD 闭环？

## Participants

PM（主持）、UXR（移动场景）、GROWTH（差异化）、ENGLD（工程边界）、BIZ（合规）。

## 边界

- **In Scope**: Server、Daemon、Mobile、Agent CLI adapter、与 popsicle pipeline 集成
- **Out of Scope**: 云端跑 Agent、legacy Yjs sync、替换 intent-coder
- **触及 charter？**: 是 — 需 **CADR** 限定为 IDD 专用派活，非通用工作流平台

---

## Phase 1: 用户需求与问题定义

### 用户痛点

离开桌面后 IDD pipeline 停滞；必须在 Cursor 手动跑 `popsicle pipeline next`。

### 目标用户

- **主要**: Popsicle IDD 开发者（本机常在线）
- **次要**: tech lead 手机审批 stage

### 约束清单

- **必须**: API Key/代码仅本机；兼容 Issue→Pipeline→Document→intent-validate
- **最好**: 首期 Cursor Agent；自托管 Server
- **可取舍**: Squad；原生 App vs PWA

### 成功指标

- 派活 60s 内 running（`[假设]`）
- spec 类 stage 无人值守完成率 ≥70%（`[待验证]`）

### 用户输入

用户确认 product 名 agent-runtime，C 档完整派活。

---

## Phase 2: 候选方案

| 方案 | 提案者 | 核心思路 |
|------|--------|---------|
| A | PM | 自建 Server+Daemon+Mobile |
| B | GROWTH | Multica 执行 + Popsicle spec |
| C | ENGLD | 仅 Daemon 调 popsicle，不调 Agent CLI |

方案 A 支持完整 delivery；B 双模型；C 无法 implement。

### 用户输入

倾向方案 A。

---

## Phase 3: 评审意见

| 角色 | 偏好 | 核心理由 |
|------|------|---------|
| PM | A | IDD 闭环一致 |
| UXR | A | 手机进度+审批 |
| GROWTH | A 长期 | Multica 仅作 P3 评估 |
| ENGLD | A | subprocess 调 popsicle |
| BIZ | A | 密钥不出本机 |

### 关键分歧

charter 张力 → CADR 限定 IDD-only；非复活 sync-collab。

---

## Phase 4: 最终决策

### PM 推荐

**方案 A**；P0 Cursor+短 pipeline；Prompt 在 daemon 读 intent-coder guide。

### 投票结果

PM/UXR/ENGLD/BIZ 支持；GROWTH 有保留地支持。

### 用户最终决策

✅ 接受 PM 推荐 → prd-writer

---

## Decision

选定 **方案 A**：新建 agent-runtime，Daemon 串联 popsicle CLI 与 Agent CLI。
