# 架构辩论纪要 — feature-arch-spec-pipeline

> PROJ-65 / ADR-029 扩展。填补「已有 product 大增量、需 ADR，但不是新 product」的 pipeline 空档。

## Topic

在 ADR-029 taxonomy 下，`feature-spec` 仅 prd→intent-spec→intent-check，缺 arch-debate/rfc/adr；`product-greenfield-spec` 含 product-debate，对 cli-ux 等已有 product 的增量过重。需新增 `feature-arch-spec`：无 facts、无 product-debate，保留完整技术 spec 链 + living-docs。

## Participants

| 角色 | 立场速写 |
|---|---|
| ARCH | 支持新增 pipeline yaml；stage 复制 migration-slice-spec 去掉 facts/debate |
| SEC | 无新运行时攻击面；仅工作流元数据 |
| PERF | 无性能影响 |
| OPS | 降低 pipeline 误选；issue-author 决策树需同步 |
| DEV | bundled intent-coder 自 heal；taxonomy 测试补一条 |

用户置信度：4/5

## Phase 1 — 技术问题 + 质量属性（NFR）

- 要解决的问题：大增量能力（如跨 crate 平台组件）需 PDR+ADR+task+intent 一次 run 定稿，但不触发 greenfield product-debate。
- 硬约束：ADR-029 `{domain}-{phase}` 命名；`feature-` 域；不破坏现有 alias。
- 质量属性优先级：可演进性 > 可用性（误选率）> 性能
- 事实基引用：PROJ-59 ADR-029；issue-author guide 反模式「增量当 greenfield」

## Phase 2 — 方案发散

- **方案 A**（ARCH）：新增 `feature-arch-spec.pipeline.yaml`，stage：prd→arch-debate→rfc→adr→intent-spec→intent-check→living-docs
- **方案 B**（DEV）：强制 arch-decision + feature-spec 两 Issue 串联 — 拒绝：traceability 碎、无 living-docs
- **方案 C**（OPS）：扩 feature-spec 可选 stage — 拒绝：pipeline 引擎无 skip，yaml 难表达

## Phase 3 — 多角色评审

| 方案 | SEC | PERF | OPS | DEV |
|---|---|---|---|---|
| A | 通过 | 通过 | 通过 | 通过 |
| B | 通过 | 通过 | 不通过 | 不通过 |
| C | 通过 | 通过 | 不通过 | 不通过 |

## Phase 4 — 收敛与决策

- ARCH 综合：方案 A 与 migration-slice-spec 结构同构、去掉迁移专属 stage，与 product-greenfield-spec 共享技术链语义。
- **用户最终决策**：采纳方案 A，canonical 名 `feature-arch-spec`。

## Decision

新增 bundled pipeline `feature-arch-spec`，归属 ADR-029 `feature-` 域；`pm-spec-only` profile 的 technical 默认保持 `feature-spec`，大增量时 Issue 显式指定 `feature-arch-spec`。

## 关键分歧

- living-docs 是否纳入：ARCH vs DEV（living-docs 增加 run 长度）→ 纳入，与 greenfield 对齐，proposed task 晋升需要。
