# 我做了两个东西：一个语言，一个运行时——让 AI 时代的开发和迁移都能对语义负责

## 开头：一个让我睡不着的问题

我手上有个老系统，近 20 万行代码（其中 10 万多行 Rust），是一个 IoT / 数字孪生平台。核心模块之一是时序数据库：设备数据写进来，先落 WAL（写前日志），进 MemTable（内存表），再刷成 Parquet Block（持久化块），用 HLC（混合逻辑时钟）做冲突检测。

这种系统最恐怖的地方不是它大，而是它**藏着一堆只有老代码知道的“怪癖”**。后来我们一条条挖出来：

- 查一个“**从没写过数据的设备**”，它不返回空结果，而是甩你一个 `Internal` 错误；
- **WAL 回放压根没接线**（源码里是个 TODO）——没来得及刷盘的数据，**崩溃就丢**；
- 写一条**时间戳更旧**的数据，它**静默丢弃**，不报错、不改状态（LWW 语义，但没写在任何文档里）。

我要把它迁到新架构。如果只是让 Copilot 照着重写，这些坑要么被原样复制，要么被 AI “顺手修好”却没人知道行为变了——**两种都是灾难。**

更长期的问题是：就算迁完了，**明天开始的日常开发**呢？新功能加进来，三个月后文档又开始骗人，新规则和老规则悄悄打架，AI 在代码库里自由发挥……

所以我没直接写代码。我做了两个东西，**一个管迁移，也管日常**：

- **intent-lang**：把规则写成逻辑，让 Z3 证明它们不矛盾；
- **Popsicle**：把“怎么做、怎么迁、怎么开发”变成人和 AI 同轨、每步可验的流水线。

先重点讲第一个，因为它是整套体系的地基。

# 一、intent-lang：一门“只说要什么，不说怎么做”的语言



## 1.1 定位：它是需求建模 DSL，不是别的

市面上的形式化工具，大多在回答“**实现对不对**”：Dafny 证明一个函数满足前后断言，TLA+ 验证一个协议不死锁，Lean 证明一个数学定理。它们都很强，但都站在“代码 / 算法已经存在”的下游。

intent-lang 站在**更上游**，回答一个几乎没人形式化过的问题：

> **我们到底想要什么、不要什么——这些要求本身，彼此矛盾吗？**

这一层长期是空白的：自然语言太模糊，TLA+ 太底层，Z3 的 SMT-LIB 又太赤裸。而在 LLM 时代，“一个可信的需求锚点”恰恰是整个开发体系里**最稀缺的资产**——因为 AI 能极快地把需求翻译成代码，却没法保证那份需求本身是自洽的。

所以 intent-lang 的定位一句话钉死：**把业务规则和领域不变量翻译成机器可读、可验证的“意图”，并用 Z3 证明多份意图之间无矛盾。** 它**不生成代码**（那是 LLM / 人 / Skill 的活），只做这一件事，做到底。

**必须先说清一条边界**：Z3 证明的是“**你写下来的规则彼此不矛盾**”，不是“你的系统是对的”。漏写了一条本该有的规则，它照样全绿——它不知道你漏了什么。换句话说，intent-lang 给的是一根**可信的需求锚点**，不是正确性的保险箱：它保证“已声明的意图自洽”，代码是否真的**符合**意图靠的是测试与 golden，意图是否**完备**永远需要人来兜。这个诚实的边界，恰恰是它值得被信任的原因——它只对它能证明的事负责。

## 1.2 七个关键字，五分钟能读懂

核心动词刻意只有 7 个（后面还有 `goal` / `coverage` / `@asis` 这些修饰性语法，但骨架就这 7 个）。下面这段是一条转账规则：

```
type Account { balance: Int }

// 全局安全规则：任何时刻余额不得为负
safety NeverNegative(a: Account) {
  invariant a.balance >= 0
}

// 一次转账操作：前置条件 + 后置条件
intent Transfer(sender: Account, amount: Int) {
  require amount > 0                                   // 前置：金额为正
  require sender.balance >= amount                    // 前置：余额够
  ensure  sender.balance' == sender.balance - amount  // 后置：x' 是执行后的值
  invariant sender.balance' >= 0                      // 前后都得成立
}
```

`type` 定义领域对象，`safety` 是全局不变量，`intent` 是一次操作，`require` / `ensure` 是前后置条件，`invariant` 前后都成立，`theorem` 表达跨规则的推论。`x'`（读作 prime）表示“执行后的值”——为了照顾 LLM，还有个等价别名 `after(x)`。

跑一下就出结论：

```
$ intent check transfer.intent
✅ safety  NeverNegative — verified
✅ intent  Transfer      — verified
```



## 1.3 设计哲学：六个关键决策

intent-lang 的每个设计都是**刻意取舍**的结果。这几个决策，基本定义了它是什么。

### 决策一：混合式——结构化语法打底，LLM 只当“草稿生成器”

做需求形式化有三条路：纯自然语言（人爱写，但同一句话 AI 能翻译出好几种矛盾的逻辑）、纯结构化语法（确定，但门槛高）、混合式。intent-lang 选了混合式，而且分工极清楚：

> **人或 LLM 用自然语言起草，翻译成结构化意图；能不能进库，由 Z3 说了算。**

这就带来了它最迷人的性质——**LLM 犯错，Z3 兜底，而且 LLM 能看着反例自己改对**：

```
// LLM 第 1 轮（有 bug：忘了检查余额）
intent Transfer(sender: Account, amount: Int) {
  require amount > 0
  ensure  sender.balance' == sender.balance - amount
  invariant sender.balance' >= 0
}
```

```
$ intent check
❌ 反例：sender.balance = 3, amount = 10
   → sender.balance' = -7，违反 invariant
```

```
// LLM 看到反例，第 2 轮自动补上一行
+ require sender.balance >= amount     → ✅ verified
```

关键在于：**反例是** `变量 = 值` **这种结构化形式，LLM 一看就懂**。换成 Lean 的 proof state 或 TLA+ 的长 trace，模型根本读不动、改不对。验证结果又是二元的（✅ 或 ❌，没有“大概对”的灰色地带），最坏情况是“验证失败”，而不是“静默出错”。这套闭环，是我认为 intent-lang 在 AI 时代最大的价值。

不过要诚实：Z3 只保证“**改完之后自洽**”，不保证 LLM 改的是**对**的那一版——把 `invariant balance' >= 0` 删掉也能变绿。机械验证挡不住“改错方向”。这正是下面第六条“反勾结”必须存在的原因：**一条因果链上，写答案的和判分的不能是同一个智能体。**

### 决策二：选 L2（SMT 求解）这个“甜点”，不多不少

形式化有四个层次，intent-lang 精准卡在第二层：


| 层次              | 代表              | 你要写      | 问题                                 |
| --------------- | --------------- | -------- | ---------------------------------- |
| L1 类型检查         | TypeScript      | 类型       | 抓不到逻辑错（`transfer(-100)` 类型没问题但逻辑错） |
| **L2 SMT 求解** ✅ | **intent-lang** | **只写条件** | **甜点：不写实现、不写证明、失败给反例**             |
| L3 模型检查         | TLA+            | 状态机      | 状态空间会爆炸                            |
| L4 定理证明         | Lean / Coq      | 定理 + 证明  | 得手写证明，曲线陡到没人用                      |


L2 的好处是：用户**不写实现、不写证明**，求解器自动求解，无界验证，失败时给反例。这和“我只想声明意图”的理念完美契合——你负责说清楚要什么，剩下的交给 Z3。

**但别神化它**：Z3 只能抓到“被你建模进**同一个可满足性问题、共享了同一批符号**”的冲突。两条规则如果自然语言上打架、形式化后却各说各话（没有共享变量或谓词），`intent check` 照样显示 verified。所以“写了 intent 就能自动发现所有冲突”是错觉——**它发现的是你恰好建模到一起的那些冲突**。这不削弱价值，但你得知道边界在哪。

### 决策三：把 `goal` 提升为一等公民，让“为什么”可被机器追踪

大项目里最常丢的信息是：“**这条 require 到底为什么存在？**” PRD 写完就忘，git log 写句 “add validation” 也回答不了。intent-lang 干脆把业务目标做成和 `safety` / `intent` 同级的语法节点：

```
goal "存储层对上层只经统一接口暴露读写" {
  rationale: "上层只依赖接口契约，不碰内部存储；契约稳则上层不受内部平移影响"
  stakeholder: ["上层业务", "SDK 开发者"]
  measure: "接口行为与 legacy 逐位等价（golden 覆盖三条路径）"
  realized_by: [RegisterTable, InsertRows, QueryLatest, ...]   // ← 反向链到意图
}
```

它不是注释，而是**能被机械检查**的结构：没有 `realized_by` 链回的 goal 会被标成“孤儿目标”；改一条意图，`intent impact` 能反查到受影响的目标和干系人。为什么不用注释？因为**注释不参与一致性证明，也不参与影响分析**。

### 决策四：`@asis` / `@tobe` 用标注表达，专治迁移的“时态”问题

迁移大型项目，必须区分“老代码**现在实际**在干什么”和“新系统**承诺**变成什么样”。intent-lang 复用已有的标注机制（不新增关键字）给意图打上时态：

```
@asis  intent FlushedBlocksSurviveRestart(...) { ... }   // 老代码就这样
@tobe  intent WalReplayRecoversCommitted(...)  { ... }   // 我们要改成这样
```

好处很实际：`intent check` 默认只验 `@tobe`，老代码可以一次性翻译进仓库而**不撑炸 CI**（`@asis` 默认跳过，`--include-asis` 才拉进来）；同一份文件能讲完整故事——“老的是这样 vs 新的应该是这样”；影响分析还能识别“哪些意图还赖在 `@asis` 老路径上没迁”。这一招，是整个迁移方法论的语言级支点（下面第三部分会真刀真枪用到）。

### 决策五：完备性检查是“沟通工具”，不是“证明工具”

“我有没有漏掉某种 角色 × 资源 × 状态 的组合？”——这是大项目最痛的完备性问题。让 Z3 去穷举所有维度会组合爆炸。intent-lang 的选择很务实：`coverage` 块声明维度的笛卡尔积，工具只在规则**文本里查每个维度值有没有出现过**，没出现就标 `uncovered`。

它接受假阳性（人一眼能识别“Admin 当然在那条规则里出现过”），因为**完备性的本质是让评审者看到“这个组合根本没人讨论过”，这已经足够有价值**，不需要动用 SMT 穷举。这体现了一以贯之的克制：能用轻办法解决沟通问题，就不上重武器。

### 决策六：反勾结原则——写意图的 AI 和写测试的 AI 必须不是同一个

这条是 AI 时代特有的纪律。如果同一个 LLM 会话既写 `.intent` 又写测试代码，它会**偷偷把两边的 bug 改成一致的**——测试和意图互相作弊，你什么都验不出来。所以规矩是：


| 阶段           | 由谁产出                     | 由谁验证   |
| ------------ | ------------------------ | ------ |
| `.intent` 草稿 | LLM A（或人）                | Z3（机械） |
| 测试代码         | **LLM B（必须 ≠ A）**        | CI     |
| 自然语言解释       | 模板（**不用 LLM**，避免二次解释丢原意） | 人      |


**核心：在任意一条因果链上，绝不让同一个智能体既写答案又判分。** 中间必须插入机械验证或独立的另一个智能体。

## 1.4 能力边界：它坚决不做的事

一门语言的克制，往往比它的能力更重要。intent-lang 明确拒绝：

- **不验时间 / 性能**：“P95 < 500ms”“最终一致”这类**禁止写进意图**——它们是 SLO，由压测和监控守；
- **不验算法实现**：“这个排序对不对”是 Dafny 的事；
- **不验协议**：“这个分布式协议会不会死锁”是 TLA+ 的事；
- **不生成代码**：那是 Skill / LLM 的活；
- **人类可读性优先，LLM 友好只是副产品**：任何“为了 AI 方便牺牲人能不能读懂”的设计，一律拒绝。

它只做一件事：**把意图变成可验证的逻辑，证明它们不矛盾。** 边界清晰，才不会变成另一个什么都想干、什么都干不好的工具。

# 二、Popsicle：一个已经能用的 IDD 运行时

先把四个名词的关系钉死，免得越读越懵：

- **IDD（意图驱动开发）**：方法论——先钉意图，再让机械验证守住下游。
- **intent-lang**：IDD 的**语言层**——上面讲的那门 DSL。
- **Popsicle**：IDD 的**运行时**——把 Issue、流水线、文档、intent 验证串成闭环的那个工具。
- **intent-coder**：跑在 Popsicle 上的**技能包**——十几个 skill + 一批 pipeline 模板，负责“抽事实 / 写 PRD / 收紧 intent / golden 对账”这些具体活。

光有意图不够。谁抽事实？PRD、RFC、ADR 放哪？什么时候允许上线？AI 下一步干嘛？Popsicle 把这条路固定下来。而且它不是一套 PPT 方法论——**是装上就能用的东西**（它自己就是用这套方法从单体一路 dogfood 迁过来的）。

## 它长什么样：一个 CLI + 一个桌面 App

**命令行（自托管 MVP）**：一条龙都在 `popsicle` 这一个二进制里——

```
popsicle init                      # 初始化 .popsicle/ 工作区
popsicle doctor --format json      # 自检：二进制 / 工作区来源对不对
popsicle issue create / start      # 建 Issue、启动一次 pipeline run
popsicle pipeline next --run <id>  # 告诉你（和 AI）下一步该干嘛
popsicle doc create / check        # 产阶段文档、拦占位符与空 checkbox
popsicle tool run intent-validate  # 对 products/ 跑 Z3 校验
```

每个命令都支持 `--format json`，**连报错都是 JSON，还带一个可执行的 `next` 字段**——这是专门给 AI Agent 用的：它不用猜下一步，直接读 `next` 往下走。

**桌面 UI（`popsicle ui`，Tauri 2）**：给人看的那一面。Issues / Pipeline / Documents / Products 四个页面，能浏览任务图、intent 块正文、Task 关系图和 intent 的 Mermaid 可视化。macOS 有 DMG：拖进 Applications、首次打开会静默把 CLI 装到 `~/.local/bin`，把 `intent` 命令和 intent-coder 技能包一起带上——**装一个 App，整套工具链就位。**

**多项目**：`~/.popsicle/global.json` 记着你所有机器 / 仓库的工作区，`popsicle project add / use / list` 一键切换，`--project` 或环境变量随时指定目标。一个人管多个仓库不用来回 `cd`。

## 它怎么存东西：过程进 DB，产品进 Git

工作区状态（Issue / Run / 文档索引）落在 `.popsicle/state.db`（SQLite）；产品的长期真相（`PRODUCT.md` / `ARCHITECTURE.md` / `intents/` / `decisions/` / `tasks/`）落在 Git 里的 `products/`。**过程性的东西进数据库，产品是什么进 Git**——这条边界让“某次 run 的临时草稿”永远不会冒充“产品的持久事实”。

## 灵魂设计：两道正交的闸

这是 Popsicle 最值钱的地方。阶段完成（`stage complete`）时，引擎跑两类**完全独立**的检查：

- **机器 gate——AI 全自动也绕不过**：真跑 `cargo test` 读退出码；golden 汇总数字必须能从明细**重算**（抓“手工编造的漂亮数字”）；意图追溯目标必须解析得到；`legacy_pin` 是占位符就**拒绝切流**。
- **人审 gate——危险动作必须人点头**：切流、改活文档这类，即便配了“代批”也要人显式确认。

**机器验“做对没”，人验“要不要放行”，两轴谁也替代不了谁。** 需要补一句诚实话：机器 gate 能保证“测试跑了、数字没编”，但**保证不了测试本身有意义**——一堆恒真的测试也能让它全绿。真正堵这个漏的是“反勾结”（测试作者 ≠ 实现作者）和变异测试，这部分靠流程纪律，不全在引擎里。别把绿灯当成质量证明，它只是“没作弊”的证明。

## 审批不是一刀切：三种模式可配

`.popsicle/project.yaml` 里配 `approval_mode`：`manual`（每步等人）/ `auto`（文档过检查就自动推进）/ `delegate-dangerous`（普通阶段代批，只有切流、改活文档这种危险阶段才必须人 `--confirm`）。**注意：机器 gate 在任何模式下都照跑**——`auto` 省的是人的点头，不是省验证。

`popsicle pipeline next` 给人和 AI 同一个导航——这是迁移和日常开发共用的引擎。下面是它两个同等重要的用法。

# 三、迁移：让老系统的每个坑都摆上台面



## 大图景：Strangler Fig，按切片绞杀

迁移不搞 Big Bang。近 20 万行的老平台按**切片**逐个迁，新旧并存，一次只搬一小块。每个切片走同一条路：

```
抽事实 → @asis 如实刻画 → 逐条决策（留 / 改） → @tobe + ADR → golden 对账 → 切流
```

行为原样平移时，还有条快车道 pipeline（`migration-preserve`），跳过完整辩论 / RFC 链。这套流程对整个平台的每个模块一视同仁——数据库、设备物模型、计费、文件服务，都是同一个套路。

## 细节：数据库切片实际怎么走

拿那个时序数据库切片当例子。对账时，端到端测试就是这么**抓出**开头那个“未知设备返回 Internal”的 bug 的。但在 IDD 里**你不能顺手改掉它**——流程是：先把它钉成 `@asis` 基线，写一份 ADR 裁定改成“存在 → 空结果、不存在 → NotFound”，落成 `@tobe`，再新增一条 golden 专门守它：

```
@tobe intent UnknownDeviceQueryReturnsEmpty(q: DeviceQuery) {
  require q.supertableRegistered
  require q.deviceWritten == false
  ensure  q.rowCount' == 0
  ensure  q.rejected'  == false     // 空，不是报错
}
```

而 LWW 静默丢弃旧写、首次写入自动建子表这类**要保留**的怪癖，就老实留在 `@asis` 不动。切流那一刻，机器 gate 校验实打实的东西：`cargo test` 全绿、golden 等价基线全绿、`legacy_pin` 是真实 commit（不是占位符），缺一不可。

**效果：凡是被测试覆盖到的行为，都没有“偷偷改掉”的空间——要么有据可查地保留，要么有据可查地修好。** 半年后有人问“当初为什么这么改”，顺着 golden → intent → ADR → fact 一路查到底，不靠任何人的记忆。

**但这里要说一句大实话**：这套方法的等价强度，**上限就是 golden 的覆盖度**。那个“未知设备返回 Internal”的 bug，是被端到端测试**测到了**才摆上台面的；一个从没被任何测试碰过的怪癖，照样会被平移过去、gate 全绿、切流通过——和“AI 原样复制”结果一样。`@asis` 是人和 AI 从近 20 万行里手抽的，抽漏了它就不知道。所以别把它当“绝对安全”，它是“**把安全边界变得可见、可审计、可持续补**”：覆盖到哪，就守到哪；没覆盖的，方法论帮你把它标成显式的盲区，而不是假装不存在。

# 四、日常开发：让新系统从第一天起就不长坑

迁移是最难的用例，但这套东西**日常开发同样是主战场**。核心是一套针对 AI 时代重新设计的产品文档哲学。

## AI 时代的 PRD：不是功能树，是任务图

传统 PRD 是一份大文档 + 一棵功能树（用户管理、计费、报表……）。死穴是：**AI 读不动、召不准**——你问一个具体问题，模型得在几万字里瞎猜。

IDD 把 PRD 拆成**任务图**：每个“用户想做的事”是一份独立 task 文件，按 **5 个固定的用户旅程阶段**归类，不许增减，也不许按 feature / 优先级 / 版本号分目录：

```
tasks/
├── onboarding/       首次成功用上某能力
├── daily-ops/        日常业务
├── troubleshooting/  出错排查
├── admin/            组织 / 配额 / 权限 / 审计
└── lifecycle/        终止 / 迁出 / 归档
```



## task 产品设计：一份 task 长什么样

这是数据库“批量写入设备数据”的真实 task：

```yaml
---
task_id: T-ST-0002                          # 稳定 ID，永不变
slug: write-device-rows                     # 可读短名，可改（git mv）
title: "我要批量写入一批设备时序数据并确认落库"   # 用户原话，不是"InsertDevice 功能"
journey_stage: daily-ops
related_intents:                            # ← 反向链到形式化意图
  - "acceptance.intent#InsertDeviceAppendsRows"
  - "invariants.intent#LwwDropsStaleWrite"
related_next_tasks: [T-ST-0003, T-ST-0004]
---

## 本 task 可解答          # ← 用户原话问句，AI 最强召回锚点
- 我怎么把一批设备数据写进去？
- 写更旧时间戳的数据会怎样？
- 写入返回成功后数据一定不会丢吗？
```

几个刻意的设计：

- **三层命名**：稳定 `T-ST-0002`（所有引用都用它）+ 可改 slug + 用户原话标题。改措辞时改标题、改 slug，**ID 永不动**，不牵连任何引用。
- **一份 ≤ 250 行**，超了说明是 epic，该拆。主路径走 happy path，异常拆到 `troubleshooting/`。
- **Chunking-friendly**：不许写“如上所述”“参考前文”（RAG 切开就失效）。每份 task 自包含，AI Copilot 能独立召回——用户问“写更旧时间戳的数据会怎样”，直接命中这份 task 和背后的 `LwwDropsStaleWrite` 意图，而不是让模型现场编。

**这就是 AI 时代 PRD 的样子：极致结构化、以任务和意图为中心、每一块都能被精准召回。**

## 设计哲学：活文档为什么能不腐烂

大项目文档烂掉的头号原因，是“一份 PRD 把商业策略、领域规则、SQL 细节混在一起”，强行把所有层拉到最高变更频率。IDD 用**七层文档 + 四条铁律**从结构上治它。按变更频率分层（L0 章程年级 → L1 领域不变量季度 → L2 产品验收 → L3 决策档案 → L4 架构契约 → L5 变更 → L6 代码），四条铁律：

1. **活文档没有版本号**，只有 `Last-Updated` + `Last-Decision-Ref`，永远代表“现在”；过期就地改，不写“曾经 / 之前”。
2. **决策档案（ADR / PDR）只追加**，Accepted 后永不就地改；错了写新决策标 `Supersedes`。
3. **每次改活文档必须引用一个 Decision ID**，CI 强制。
4. 每份 ADR 必须声明它动了哪层 intent——**CI 拒绝缺这项的提交**。

于是“为什么这么设计”永远查得到，文档永远只描述现在，历史全在只追加的决策档案里。

有人会立刻反问：迁移不就是“从旧到新”的历史吗，这跟“活文档不写历史”不打架？——不打架，正好是分工：**活文档只说“现在的系统是什么样”**，而“从 X 迁到 Y 的过程、为什么这么迁”属于历史，进 ADR 和一张专门的 traceability 对照表。活文档保持干净，历史一条不丢，各归各位。

## 最大的日常收益：加新功能 = 先证明它不和老规则打架

比如给数据库加一条“冷数据自动归档”规则。传统做法直接写代码，等它和“查询必须覆盖历史区间”冲突时线上暴雷。IDD 里你先写一条意图，`intent check` 当场告诉你：它和已有规则矛盾吗（前提是两条规则建模时共享了符号）？**能被它抓到的矛盾，会在编译前暴露，不用等事故后复盘。** 新功能同样按场景走 pipeline（`feature-spec` → `feature-delivery` 做增量、`product-greenfield-spec` 起新产品、`fix-regression` 修回归），每一步同样过机器 gate 和文档检查。

# 五、什么时候**别**用它

这套东西不是银弹，代价也很实在：一个切片要走 `抽事实 → 辩论 → PRD → RFC → ADR → intent → 实现 → golden → 切流`，前期 overhead 不小。所以我自己的判断是：

- **改一两行的 bug、一次性脚本、周末小项目**——别用，`fix-regression` 都嫌重，直接写。
- **需求还在剧烈漂移、朝令夕改的探索期**——别急着形式化，意图会跟着改到崩溃；等规则稳下来再钉。
- **纯算法 / 协议正确性**——用错工具了，那是 Dafny / TLA+ 的地盘。
- **团队没人愿意维护活文档和决策档案**——工具能强制引用 Decision ID，但强制不了人认真写；没有这个共识，它会退化成一堆走过场的空壳。

它真正划算的场景是：**长期演进的中大型系统、遗留迁移、以及“人 + 多个 AI Agent 长期协作同一套代码”**——也就是“语义会被慢慢改坏、而没人守得住”的地方。规模越大、活得越久、协作的手越多，这套 overhead 摊下来越值。

## 迁移和日常都怕的三句自欺，这套方法各治一句


| 自欺之词       | 治它的东西                                                       |
| ---------- | ----------------------------------------------------------- |
| “感觉差不多”    | golden 对账（数字对不上切不了流）                                        |
| “文档写了就行”   | Z3 判一致性（`@asis` / `@tobe`、新规则先证不矛盾）                         |
| “AI 已经写完了” | 机器 gate（`cargo test`、`legacy_pin`、重算校验，`auto` 绕不过）+ 反勾结（写测试的 ≠ 写实现的） |

## 说到底，我做了件什么事

> **我做了一套“让 AI 时代还能对语义负责”的工具链：**
>
> **intent-lang** 把“要什么”编译成 Z3 能判定的逻辑——一门只说要什么、不说怎么做，LLM 能生成、Z3 能兜底的需求建模语言；
>
> **Popsicle** 把“怎么做、怎么迁、怎么开发”变成人和 AI 同轨、每步机器可验的流水线。
>
> 它一手托两件事——**迁移**时让老系统的每个坑摆上台面，由人裁定留还是改、由机器证明确实做到；**日常开发**时用任务图 + 七层文档 + Z3，让新系统从第一天起就不长新坑。

大模型让“写代码”变便宜了，但**“写对”不会自动变便宜**。迁移面对的是会假装报错、会悄悄丢数据的历史行为；日常开发面对的是“三个月后文档就骗人”的老问题。这两件事，缺的是同一根**可信锚点**。

如果你也在啃遗留系统，或者只是不想让 AI 在代码库里自由发挥——从最小一步开始：把一条真实行为写成 `@asis` 意图，跑一次 `intent check`。第一次看见“假装报错”的 bug 被 golden diff 标红，或者一条新规则被 Z3 当场判矛盾时，你就懂了。
